# 需求分析报告 — 《真言地牢》游戏设计评估

## 一、显性需求（用户直接提出）

### 1.1 设计分析（P0）
- 全面审阅现有 GDD/PRD v1.0 文档的 18 个章节内容
- 评估设计完整性、逻辑自洽性、技术可行性
- 识别各系统之间的耦合关系和潜在冲突点
- 评估核心体验闭环是否成立（"输入单词打怪"的爽感 + "学习效果"的平衡）

### 1.2 完善建议（P0）
- 针对文档中缺失或不足的模块提出具体补充方案
- 针对设计中存在逻辑漏洞或可行性风险的模块提出修正建议
- 针对"需求定稿"标记但实际未达到交付标准的部分指出问题
- 补充缺失的商业化、运营、增长、社交等维度设计

### 1.3 开发到上线阶段规划（P0）
- 基于文档已有的 4 个里程碑进行深化和修正
- 补充里程碑 0（立项/预研）和里程碑 5（测试/调优/上线）
- 每个阶段明确交付物、验收标准、人员分工、工期估算
- 包含风险管理与缓冲时间

### 1.4 策略建议（P0）
- 技术选型策略（Unity vs Cocos 的决策依据细化）
- 团队组建策略（各阶段人员配置与角色要求）
- 测试与质量保障策略
- 上线与运营策略
- 用户获取与增长策略

## 二、隐性需求（基于文档内容推断）

### 2.1 文档质量与完整性问题（P0）

**缺失的系统模块**：
- 商业化/变现系统设计（付费点、IAP、订阅、广告位）
- 社交系统（好友、排行榜、协作/对战模式、分享）
- 云存档与跨平台同步（多平台目标但未提数据同步方案）
- 热更新方案（8000 词库 + 内容更新的持续交付机制）
- 数据埋点与数据分析体系（仅提及四个字，无具体事件定义）
- 反作弊与数据安全（客户端存档易被篡改）
- 客服与反馈系统
- A/B 测试框架设计
- 叙事与世界观深度设计（仅有 Elevator Pitch，无剧情线、NPC 设定、结局设计）
- 音效/美术具体资源清单（Asset List 缺失）

**设计逻辑潜在问题**：
- 8000 词全部手工标注 theme、game_type、weakness_tags、combo_words 的可行性存疑
- 词向量余弦相似度阈值 0.35 的语义一致性标准过于宽松，可能导致主题涣散
- 语音模块支持完全离线（Whisper.cpp）但 iOS/Android 的模型加载性能未验证
- 三词禁咒（3 秒内输入 3 词）在移动端的操作可行性存疑
- DDA 参数缺少基准数据支撑（连续 3 次错误降难度、连续 5 次秒杀提难度的数值缺乏依据）
- 学习效果闭环缺少教育效果验证方案（如何证明玩家真的学会了单词）

**技术可行性风险**：
- SQLite 在移动端查询 8000 词 + 语义边表的性能需要基准测试
- Whisper.cpp base.en 在低端 Android 的实时推理性能未验证
- WebGL 平台的语音不可用导致核心体验缺失（降级方案不足）
- 内存占用目标 500MB 含 Whisper 模型，但加上 Unity 运行时、2D URP、资源后接近上限

### 2.2 多平台适配需求（P0）
- PC/Mac/iOS/Android/WebGL 五平台适配的技术差异与资源复用策略
- 各平台输入方式差异（键盘/虚拟键盘/语音/手柄）的 UX 统一性
- 各平台审核要求（App Store、Google Play、Steam）的差异与合规准备
- 各平台性能分级与画质档位设计

### 2.3 运营与增长需求（P1）
- 冷启动策略（种子用户获取、核心社区建设）
- 内容运营（每日挑战、主题活动、词库更新节奏）
- 用户留存体系（Push 通知、召回机制、学习打卡激励）
- 口碑传播设计（分享战绩、学习进度可视化、社交货币）
- 与教育机构/KOL 的合作路径

### 2.4 教育属性合规需求（P1）
- 教育类游戏的年龄分级与内容审核要求（各平台差异）
- 面向未成年人的隐私保护合规（COPPA、GDPR-K、中国未成年人保护法）
- 学习效果声称的法律风险（避免被认定为"未经验证的教育效果"）
- 数据收集与使用的用户授权机制

### 2.5 持续迭代需求（P1）
- 版本迭代节奏（大版本/小版本/热更新的频率与内容规划）
- 用户反馈闭环（从收集到设计到验证的流程）
- 内容扩展计划（8000 词之后的词库扩展、新主题、新机制）

## 三、交付物需求

### 3.1 报告格式（P0）
- Markdown 格式，便于团队内部流转与版本管理
- 结构清晰，支持按模块快速定位
- 每个章节有明确的优先级标注和可操作的建议项

### 3.2 语言与深度（P0）
- 中文撰写，技术术语保留英文原文
- 深度达到可直接转化为任务单的程度（包含具体参数建议、接口设计思路、资源配置方案）
- 区分"设计层建议"（策划做什么）与"实现层建议"（程序怎么实现）

### 3.3 可操作性要求（P0）
- 每条建议附带实施优先级（P0/P1/P2）
- 每条建议附带 estimated effort（粗略工期估算）
- 建议之间标注依赖关系（哪些必须先完成）
- 提供可供直接参考的竞品案例或技术方案

### 3.4 完整性要求（P1）
- 覆盖 GDD/PRD 全部 18 个章节的逐章评估
- 覆盖文档未涉及的 8 个新增维度（商业化、社交、运营、叙事、合规、持续迭代、数据、资源管理）
- 提供从 Day 0 到上线后 90 天的完整时间线

## 四、目标读者分析

### 4.1 游戏制作人/项目经理（P0 受众）
- 需要的信息：整体可行性判断、工期与资源估算、风险清单、里程碑规划
- 信息粒度：高层摘要 + 关键决策点 + 资源需求
- 使用场景：立项评审、预算申请、进度管控

### 4.2 策划团队（P0 受众）
- 需要的信息：各系统设计漏洞与修正建议、缺失模块的完整设计方案、数值平衡建议、体验优化方向
- 信息粒度：具体参数、设计替代方案、玩家体验影响分析
- 使用场景：设计迭代、文档补充、玩法调优

### 4.3 程序团队（P0 受众）
- 需要的信息：技术风险评估、架构建议、性能优化方向、关键技术选型决策、第三方库推荐
- 信息粒度：技术方案对比、伪代码级建议、性能基准参考值
- 使用场景：技术预研、架构设计、编码实现

### 4.4 美术团队（P1 受众）
- 需要的信息：风格统一性建议、资源量估算、美术管线优化建议、多平台适配的美术策略
- 信息粒度：具体预制体清单、风格参考图优先级、分辨率与格式规范
- 使用场景：美术预研、资源生产、管线搭建

### 4.5 运营/发行团队（P1 受众）
- 需要的信息：上线节奏、平台策略、用户获取路径、商业化方案、数据分析体系
- 信息粒度：可执行的活动方案、数据指标体系、渠道选择建议
- 使用场景：发行准备、运营规划、市场推广

### 4.6 投资人/管理层（P2 受众）
- 需要的信息：市场机会判断、竞争优势分析、投资回报预期、风险评估
- 信息粒度：高层战略摘要、竞品对标、商业化潜力
- 使用场景：投资决策、战略方向确认
